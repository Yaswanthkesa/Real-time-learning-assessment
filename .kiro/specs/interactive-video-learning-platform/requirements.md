# Requirements Document

## Introduction

The Interactive Video Learning Platform is a MERN stack web application that enables teachers to upload educational videos and students to watch them with AI-generated multiple-choice questions (MCQs) that pause the video. The system uses Whisper for video transcription and Mistral-7B-Instruct LLM to generate concept-based MCQs. Wrong answers cause the video to replay from the current concept's timestamp, creating an adaptive learning experience where students must correctly answer each MCQ before progressing. Authentication supports both email/password and Google OAuth.

## Glossary

- **System**: The Interactive Video Learning Platform web application
- **Teacher**: A registered user with permission to create courses and upload videos
- **Student**: A registered user with permission to enroll in courses and watch videos
- **Course**: A collection of educational videos created by a teacher with metadata (name, description, category, difficulty)
- **Video**: An educational video file uploaded by a teacher to a course
- **MCQ**: Multiple-choice question with 4 options and 1 correct answer
- **Concept**: A key learning topic extracted from video transcription
- **Transcription**: Text representation of video audio generated by Whisper
- **Course_ID**: Auto-generated unique identifier for a course
- **Thumbnail**: An image extracted from the video to represent it visually
- **Video_Processor**: The subsystem that handles video transcription and MCQ generation
- **Auth_System**: The subsystem that handles user authentication and authorization
- **Progress_Tracker**: The subsystem that tracks student video progress and MCQ performance
- **Dashboard**: The teacher interface for viewing analytics

## Requirements

### Requirement 1: User Registration and Authentication

**User Story:** As a teacher or student, I want to register and log in with email/password or Google OAuth, so that I can access the platform securely.

#### Acceptance Criteria

1. THE Auth_System SHALL provide separate registration for teachers and students
2. THE Auth_System SHALL support two authentication methods: email/password and Google OAuth
3. WHEN a user registers with email/password, THE Auth_System SHALL require email, password, name, and role (teacher or student)
4. WHEN a user registers, THE Auth_System SHALL validate email format and password strength (minimum 8 characters)
5. WHEN a user logs in with Google OAuth, THE Auth_System SHALL retrieve user profile from Google and create/update user account
6. WHEN a user logs in with valid credentials, THE Auth_System SHALL generate a JWT token valid for 7 days
7. WHEN a user logs in with invalid credentials, THE Auth_System SHALL return an authentication error
8. THE Auth_System SHALL store passwords using bcrypt hashing
9. THE Auth_System SHALL prevent duplicate email registrations
10. WHEN a user logs in with Google OAuth for the first time, THE Auth_System SHALL prompt for role selection (teacher or student)

### Requirement 2: User Profile Management

**User Story:** As a user, I want to manage my profile information, so that I can keep my account details current.

#### Acceptance Criteria

1. THE System SHALL store user profiles with name, email, profile picture, and bio
2. WHEN a user updates their profile, THE System SHALL validate and save the changes
3. THE System SHALL allow users to upload profile pictures in JPEG or PNG format
4. WHEN a profile picture is uploaded, THE System SHALL validate file size does not exceed 5MB

### Requirement 3: Course Creation

**User Story:** As a teacher, I want to create courses with descriptive information, so that students can find and understand my courses.

#### Acceptance Criteria

1. WHEN a teacher creates a course, THE System SHALL require course name, description, category, and difficulty level
2. WHEN a course is created, THE System SHALL auto-generate a unique Course_ID
3. THE System SHALL validate course name is between 3 and 100 characters
4. THE System SHALL validate description is between 10 and 1000 characters
5. THE System SHALL support difficulty levels: beginner, intermediate, and advanced
6. THE System SHALL associate each course with the creating teacher's user ID

### Requirement 4: Video Upload and Storage

**User Story:** As a teacher, I want to upload multiple videos to my course, so that I can provide comprehensive educational content.

#### Acceptance Criteria

1. WHEN a teacher uploads a video, THE System SHALL accept MP4, AVI, and MOV formats
2. WHEN a video is uploaded, THE System SHALL validate file size does not exceed 500MB
3. THE System SHALL store uploaded videos on the server filesystem
4. WHEN a video is uploaded, THE System SHALL extract a thumbnail image from the video
5. THE System SHALL allow multiple videos per course
6. THE System SHALL store video metadata including filename, duration, upload timestamp, and order
7. WHEN videos are displayed, THE System SHALL order them by upload sequence

### Requirement 5: Video Transcription

**User Story:** As a student, I want videos to have accurate transcriptions, so that MCQs can be generated from the content.

#### Acceptance Criteria

1. WHEN a student first watches a video, THE Video_Processor SHALL transcribe the audio using Whisper
2. THE Video_Processor SHALL generate transcription segments with start time, end time, and text
3. THE Video_Processor SHALL store the complete transcription for future use
4. IF transcription fails, THEN THE System SHALL log the error and notify the student
5. THE Video_Processor SHALL process transcription within 2 minutes for videos up to 10 minutes long

### Requirement 6: MCQ Generation from Transcription

**User Story:** As a student, I want MCQs generated from video content, so that I can test my understanding while watching.

#### Acceptance Criteria

1. WHEN transcription completes, THE Video_Processor SHALL extract 3 to 5 key learning concepts using an LLM
2. FOR EACH concept extracted, THE Video_Processor SHALL generate one MCQ with 4 options labeled A, B, C, D
3. THE Video_Processor SHALL validate each MCQ has exactly 4 options and 1 correct answer
4. THE Video_Processor SHALL map each concept to a timestamp in the video based on keyword matching
5. IF keyword matching fails for a concept, THEN THE Video_Processor SHALL distribute concepts evenly across video duration
6. THE Video_Processor SHALL store MCQs with concept title, question text, options array, correct answer, and timestamp
7. THE Video_Processor SHALL sort MCQs by timestamp in ascending order

### Requirement 7: MCQ Display During Video Playback

**User Story:** As a student, I want MCQs to appear as overlays during video playback, so that I can answer questions without leaving the video.

#### Acceptance Criteria

1. WHEN video playback reaches an MCQ timestamp, THE System SHALL pause the video
2. WHEN the video pauses for an MCQ, THE System SHALL display a black overlay on the video player
3. THE System SHALL display the concept title, question text, and 4 options on the overlay
4. THE System SHALL allow the student to select one option
5. WHEN a student selects an option, THE System SHALL highlight the selected option
6. THE System SHALL provide a submit button to confirm the answer

### Requirement 8: MCQ Answer Validation and Concept Replay Logic

**User Story:** As a student, I want immediate feedback on my answers, so that I know if I need to rewatch the content.

#### Acceptance Criteria

1. WHEN a student submits an MCQ answer, THE System SHALL compare it to the correct answer
2. IF the answer is correct, THEN THE System SHALL remove the overlay and resume video playback to the next concept
3. IF the answer is incorrect, THEN THE System SHALL display an error message with the correct answer
4. WHEN an incorrect answer is submitted, THE System SHALL restart video playback from the current concept's timestamp (not from the beginning)
5. THE System SHALL keep the student on the same MCQ until they answer correctly
6. THE System SHALL track the number of attempts for each MCQ per student
7. THE System SHALL allow unlimited attempts for each MCQ until the correct answer is provided

### Requirement 9: Course Search and Discovery

**User Story:** As a student, I want to search for courses by name or ID, so that I can find relevant educational content.

#### Acceptance Criteria

1. THE System SHALL provide a search interface accepting course name or Course_ID as input
2. WHEN a student searches by course name, THE System SHALL return all courses with names containing the search term (case-insensitive)
3. WHEN a student searches by Course_ID, THE System SHALL return the exact course matching that ID
4. THE System SHALL display search results showing course name, teacher name, category, difficulty, and thumbnail
5. THE System SHALL allow multiple teachers to create courses with the same name
6. WHEN search results are displayed, THE System SHALL show all matching courses regardless of teacher

### Requirement 10: Video Playback Sequence Enforcement

**User Story:** As a student, I want to watch videos in the intended order, so that I learn concepts progressively.

#### Acceptance Criteria

1. THE System SHALL enforce sequential video watching within a course
2. WHEN a student accesses a course, THE System SHALL only allow playback of the first unwatched video
3. THE System SHALL mark a video as watched when all MCQs are answered correctly
4. THE System SHALL unlock the next video when the current video is marked as watched
5. THE System SHALL display locked status for videos not yet accessible

### Requirement 11: Student Progress Tracking

**User Story:** As a student, I want to see my progress in each course, so that I can track my learning journey.

#### Acceptance Criteria

1. THE Progress_Tracker SHALL calculate course completion percentage as (watched videos / total videos) × 100
2. THE Progress_Tracker SHALL store MCQ scores for each student per video
3. THE Progress_Tracker SHALL track total time spent on each video per student
4. WHEN a student views their progress, THE System SHALL display completion percentage, MCQ scores, and time spent
5. THE Progress_Tracker SHALL update progress in real-time as students complete videos

### Requirement 12: Teacher Analytics Dashboard

**User Story:** As a teacher, I want to view student analytics for my courses, so that I can understand student engagement and performance.

#### Acceptance Criteria

1. THE Dashboard SHALL display total student enrollment count per course
2. THE Dashboard SHALL display MCQ performance statistics including average score and question-level accuracy
3. WHEN a teacher views analytics, THE System SHALL aggregate data across all enrolled students
4. THE Dashboard SHALL allow teachers to filter analytics by course
5. THE Dashboard SHALL display student-level MCQ scores showing individual performance

### Requirement 13: Authentication Middleware

**User Story:** As a system administrator, I want protected routes to verify user authentication, so that unauthorized users cannot access restricted features.

#### Acceptance Criteria

1. WHEN a protected route is accessed, THE Auth_System SHALL verify the JWT token from the Authorization header
2. IF no token is provided, THEN THE Auth_System SHALL return a 401 unauthorized error
3. IF an invalid token is provided, THEN THE Auth_System SHALL return a 401 unauthorized error
4. WHEN a valid token is verified, THE Auth_System SHALL extract user ID and role and attach them to the request
5. THE Auth_System SHALL allow the request to proceed to the route handler when authentication succeeds

### Requirement 14: Role-Based Access Control

**User Story:** As a system administrator, I want to restrict features based on user roles, so that teachers and students have appropriate permissions.

#### Acceptance Criteria

1. THE System SHALL restrict course creation to users with teacher role
2. THE System SHALL restrict video upload to users with teacher role
3. THE System SHALL restrict analytics dashboard access to users with teacher role
4. THE System SHALL allow course enrollment and video watching for users with student role
5. WHEN a user attempts an unauthorized action, THE System SHALL return a 403 forbidden error

### Requirement 15: Video Processing Error Handling

**User Story:** As a student, I want clear error messages when video processing fails, so that I understand what went wrong.

#### Acceptance Criteria

1. IF video transcription fails, THEN THE System SHALL log the error with video ID and timestamp
2. IF MCQ generation fails, THEN THE System SHALL log the error and retry up to 2 times
3. IF video processing fails after retries, THEN THE System SHALL notify the student with a descriptive error message
4. THE System SHALL allow students to retry video processing manually
5. THE System SHALL store processing status (pending, processing, completed, failed) for each video

### Requirement 16: Data Persistence and Integrity

**User Story:** As a system administrator, I want all data stored reliably in MongoDB, so that user data and course content are not lost.

#### Acceptance Criteria

1. THE System SHALL store user accounts in a MongoDB users collection
2. THE System SHALL store courses in a MongoDB courses collection with teacher reference
3. THE System SHALL store videos in a MongoDB videos collection with course reference
4. THE System SHALL store MCQs in a MongoDB mcqs collection with video reference
5. THE System SHALL store student progress in a MongoDB progress collection with student and video references
6. THE System SHALL enforce referential integrity using MongoDB ObjectId references
7. WHEN a course is deleted, THE System SHALL cascade delete all associated videos and MCQs
