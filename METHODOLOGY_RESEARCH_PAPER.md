# 3. METHODOLOGY

## 3.1 System Overview

The proposed system implements an end-to-end machine learning pipeline for automated generation of interactive multiple-choice questions from educational video content. The architecture comprises three primary components: a React-based presentation layer for user interaction, a Node.js application layer for orchestration and business logic, and a Python-based machine learning pipeline for content analysis and question generation. Data persistence is achieved through MongoDB, which stores video metadata, time-aligned transcripts, and generated assessments. The system supports three processing modes—Fast, Cloud, and Full—to accommodate varying computational resources and latency requirements. The Fast mode utilizes Whisper tiny with Flan-T5-base for rapid processing, the Cloud mode leverages Groq API with Llama 3.3 70B for scalable inference, and the Full mode employs Whisper base with Mistral-7B-Instruct for maximum accuracy in local deployments.

## 3.2 Data Acquisition and Preprocessing

Educational videos are uploaded through a web interface with validation for standard formats (MP4, AVI, MOV) and size constraints (500MB maximum). Upon successful validation, videos are stored in a structured filesystem hierarchy while metadata is persisted in MongoDB with an initial processing status. Audio extraction is performed using FFmpeg, which converts the video audio stream to monaural waveform sampled at 16 kilohertz. This sampling rate represents the optimal input specification for Whisper models, which were trained on 16 kHz audio data. The conversion command specifies single-channel audio to reduce computational complexity while preserving speech intelligibility, as stereo information provides minimal benefit for speech recognition tasks. The extracted audio is temporarily stored as a WAV file with linear PCM encoding to ensure lossless representation during subsequent processing stages. Error handling mechanisms manage common failure scenarios including corrupted files and unsupported codecs, with failures properly logged and communicated to users. Temporary audio files are automatically deleted after successful transcription to conserve storage resources.

## 3.3 Speech-to-Text Transcription Model

The automatic speech recognition component employs OpenAI Whisper, a transformer-based encoder-decoder architecture trained on 680,000 hours of multilingual supervised data. The system utilizes the base model variant containing 74 million parameters, which provides an optimal balance between transcription accuracy and computational efficiency. The base model achieves word error rates comparable to commercial systems while maintaining reasonable inference times on consumer hardware. For scenarios requiring faster processing, the tiny model variant with 39 million parameters reduces processing time by approximately 60 percent while maintaining sufficient accuracy for educational content.

Whisper implements a sequence-to-sequence architecture where the encoder processes input audio spectrograms through multiple transformer layers to generate contextual representations, and the decoder autoregressively generates transcript text conditioned on these representations. The model was trained using a multitask learning objective encompassing speech recognition, language identification, timestamp prediction, and voice activity detection. This multitask training enables production of time-aligned transcripts without requiring additional post-processing alignment algorithms, which is critical for synchronizing questions with video playback positions.

The transcription process implements model caching to avoid redundant loading operations when processing multiple videos. Models are loaded once during the first processing request and retained in memory for subsequent requests, reducing initialization overhead from 3-5 seconds per video to a one-time cost. The audio file is passed to the model's transcribe method, which internally performs mel-spectrogram computation, encoder inference, and decoder inference with beam search decoding. The system configures transcription with language specification set to English and task type set to transcribe, ensuring optimization for accurate English transcription. The output consists of time-aligned segments, each representing a semantically coherent utterance with start and end timestamps specified in seconds with millisecond precision. This temporal alignment enables precise synchronization between video playback and question presentation.

## 3.4 MCQ Generation using Language Models

The question generation component leverages large language models to transform transcript text into structured educational assessments. The system employs Mistral-7B-Instruct-v0.2 as the primary model, a 7-billion-parameter transformer decoder fine-tuned for instruction-following tasks. Mistral-7B was selected for its superior performance on reasoning benchmarks, efficient grouped-query attention mechanism that reduces memory requirements, and permissive licensing enabling educational deployment. The model implements sliding window attention that enables processing of longer context windows while maintaining computational efficiency, which is beneficial for processing lengthy educational transcripts.

The question generation process employs prompt engineering to elicit structured output from the language model. The prompt instructs the model to identify three to five key learning concepts from the provided transcript, generate one multiple-choice question per concept with exactly four options labeled A through D, and return output in valid JSON format. An example output structure is included in the prompt to leverage the model's few-shot learning capabilities and improve output consistency. Transcript text is truncated to 3000 characters before prompt inclusion to ensure the combined prompt and expected output remain within the model's context window while preserving salient content.

The language model is configured with a temperature parameter of 0.4, which reduces randomness in token selection and promotes deterministic outputs while maintaining sufficient diversity to generate varied questions across different videos. The maximum token limit is set to 1024 to accommodate generation of multiple questions with explanations. The model is loaded with 16-bit floating-point precision to reduce memory requirements by 50 percent while maintaining negligible impact on output quality, enabling deployment on systems with limited GPU memory.

For deployment scenarios where local inference is impractical, the system provides an alternative implementation using Groq API with Llama 3.3 70B model. This cloud-based approach offloads computational requirements to specialized inference infrastructure while maintaining consistent output formats. The system implements retry logic with exponential backoff to handle transient API failures and includes fallback question generation that produces structurally valid questions when primary methods fail.

The language model output undergoes validation and parsing to extract structured question data. Regular expression matching locates JSON array structures within model output, as language models occasionally generate explanatory text surrounding the requested JSON. Extracted JSON is parsed and validated to ensure each question contains required fields including concept title, question text, exactly four options, and correct answer designation. Questions failing validation are discarded to maintain output quality.

## 3.5 Timestamp Mapping Algorithm

The timestamp mapping algorithm establishes temporal associations between generated questions and relevant video segments. The algorithm implements a keyword-based semantic matching approach that identifies transcript segments discussing concepts related to each question. The mapping process begins by extracting keywords from each question's concept title through tokenization and filtering to retain only words exceeding three characters in length. This threshold eliminates common function words while preserving content words likely to appear in topically relevant segments. Filtered keywords are converted to lowercase and stored in a set structure for efficient membership testing.

For each question, the algorithm iterates through transcript segments in temporal order and computes keyword overlap between concept keywords and segment words. When a segment containing one or more concept keywords is found, that segment is selected as the temporal anchor and the question is assigned the segment's start timestamp. This first-match strategy prioritizes earlier concept occurrences, aligning with pedagogical structures where concepts are introduced before elaboration.

When no transcript segment contains concept keywords—which can occur due to paraphrased terminology or transcription errors—the algorithm employs a fallback strategy that distributes questions evenly across video duration. This approach divides video duration into equal intervals based on question count and assigns each question to its corresponding interval midpoint. After timestamp assignment, questions are sorted in ascending temporal order to ensure logical sequencing during playback.

## 3.6 System Integration and Backend Processing

The backend implements a service-oriented architecture that orchestrates the machine learning pipeline while maintaining separation between web service responsibilities and computational tasks. Integration between Node.js backend and Python machine learning pipeline is achieved through subprocess spawning with inter-process communication via standard streams. When processing is triggered, the backend constructs a command-line invocation of the Python script with video file path and audio output path as arguments. The spawn function creates a Python process and establishes bidirectional communication channels, capturing standard output to receive structured JSON results while monitoring standard error for progress messages and diagnostics.

The Python script is designed as a self-contained module that accepts command-line arguments, performs all processing stages sequentially, and emits a single JSON result to standard output. Error handling is implemented at each stage to catch exceptions and return structured error responses. The backend implements comprehensive error handling to manage failures at any pipeline stage, updating video processing status appropriately and storing error messages for user visibility.

Upon successful pipeline completion, the backend persists results to MongoDB through Mongoose object-document mapping. Transcript segments are stored in a Transcript collection with references to parent video documents, while each multiple-choice question is stored as a separate document in an MCQ collection with fields for question text, options, correct answer, explanation, concept title, timestamp, and video reference. This normalized schema enables efficient querying by video, timestamp range, or concept.

## 3.7 Frontend Interaction and User Experience

The frontend implements an interactive video player using React Player, which provides consistent interface across video formats while exposing programmatic playback controls. The player component maintains state for current playback position, play/pause status, and active question display through React hooks managing synchronization between video progress and question presentation.

The question overlay mechanism monitors playback position through periodic polling that queries the player's current time at regular intervals. When playback position crosses a question's assigned timestamp, the system triggers question display by updating component state to show the overlay and pausing video playback. The overlay is rendered as a modal component with semi-transparent backdrop, presenting question text and four options with radio button controls.

The interaction workflow implements immediate feedback mechanisms. When students submit answers, the system immediately indicates correctness through visual styling, reveals the correct answer, and displays explanatory text. Progress tracking captures detailed analytics, with the frontend periodically sending progress updates to the backend at five-second intervals, reporting playback position and watched duration. Question responses are immediately transmitted with question identifier, selected answer, and correctness status, enabling computation of per-video and per-course performance statistics.

## 3.8 Performance Optimization Techniques

Database query performance is optimized through strategic index creation on frequently queried fields including video identifiers, student identifiers, and processing status values. MongoDB compound indexes on field combinations appearing together in query predicates enable efficient document location without full collection scans, reducing query latency from hundreds of milliseconds to single-digit milliseconds.

The machine learning pipeline implements model caching to amortize loading costs across multiple processing requests. Video streaming utilizes HTTP range requests to enable efficient progressive loading and seeking operations, with the backend examining Range headers to determine which byte ranges should be transmitted. Frontend code splitting and lazy loading reduce initial page load times, with React components loaded on-demand using dynamic imports wrapped in React.lazy, reducing initial bundle size from approximately 800 kilobytes to 200 kilobytes.

## 3.9 Evaluation Strategy

System evaluation encompasses transcription accuracy, question quality, timestamp alignment precision, and usability. Transcription accuracy is assessed by comparing Whisper-generated transcripts against human-annotated ground truth using word error rate metrics. Question quality evaluation employs both automated metrics including syntactic validity checking and semantic coherence scoring, and human expert assessment rating questions on relevance, clarity, difficulty appropriateness, and distractor quality. Timestamp alignment accuracy is evaluated through human annotation of alignment quality classifications. System usability is assessed through user studies measuring task completion rates, time-on-task, and subjective satisfaction using standardized instruments such as the System Usability Scale.
